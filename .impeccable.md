# 月海 Codex 壁纸设计与注入边界

> 本文件是制作、评审和修改月海皮肤前的必读约定。它记录当前真实实现，不替代 `PRODUCT.md`；产品定位以 `PRODUCT.md` 为准，具体数值以对应源码为准。

## 1. 设计语境

### 用户

- 面向想更换 Codex 外观、但不想理解脚本、客户端结构和注入细节的普通用户。
- 同时支持 Windows 与 macOS，最终体验必须是一键应用、立即生效、可以恢复。

### 核心任务

- 普通壁纸：免费渐变壁纸，使用完整月海运行时。
- Pro 壁纸：精制图片壁纸，使用同一套月海运行时并作为付费业务层。
- 任何皮肤都不能破坏聊天、代码审查、终端、弹层、输入框和键盘操作的可读性。

### 视觉气质

- 安静、清透、连贯、有月海感；视觉服务于工作，不抢内容注意力。
- 冷色中性底色为主，青蓝只用于主操作、选中态与连接状态。
- 同一连续平面必须共享同一个表面令牌。禁止用多个“看起来差不多”的透明度手工调平。
- 控件可以比环境表面更实，但必须是为了可读性和操作识别，不是装饰性玻璃效果。
- 禁止通用 SaaS 仪表盘感、无意义控件、假进度和满屏玻璃卡片。

### 质量标准

- 按可发布皮肤处理，不按临时 Demo 处理。
- 正文和控件至少达到 WCAG AA；键盘焦点必须可见。
- Codex 官方 DOM 或资源结构变化时必须明确失败，禁止静默套用不完整皮肤。

## 2. 技术统一，业务分级

| 类型 | 背景来源 | 运行时能力 | 业务含义 |
| --- | --- | --- | --- |
| 普通壁纸 `standard` | CSS 渐变 | 透明表面、正文增强、月海助手、交互特效、自定义壁纸 | 免费基础款 |
| Pro 壁纸 `pro` | 精制图片 | 与普通壁纸完全一致 | 付费精选款 |

**边界规则：** 两类壁纸必须走同一条月海 CSS/JS 运行时路线。只允许调用 Codex 官方 `set_mode` 同步明暗结构；禁止再通过官方 `set_theme` 传入颜色。`edition` 只表达业务层级，不表达技术能力。

## 3. 当前四层改造能力

### A. 壁纸运行时配置：所有壁纸共用

源码：`src/theme-catalog.mjs`、`theme/runtime/appearance-bridge.template.js`

| 字段 | 当前含义 | 约束 |
| --- | --- | --- |
| `mode` | `light` 或 `dark` | 只用于同步 Codex 明暗结构 |
| `tier` | `standard` 或 `pro` | 必须与主题 `edition` 一致 |
| `backgroundGradient` | 普通壁纸背景 | 必须与官网卡片 `previewGradient` 一致 |
| `wallpaper` | Pro 图片文件 | 只能引用安装包内受控资源 |
| `wallpaperGradient` | Pro 图片遮罩 | 保证文字与界面边缘可读 |
| `palette` | 完整月海界面令牌 | 控制文字、表面、边框、状态和保护层 |

### B. CSS 令牌覆盖：统一表面系统

源码：`theme/static/theme.css`

- 通过 `html.codex-moonsea` 作为总开关。
- 先定义 `--moonsea-*` 主题令牌，再映射到 Codex 的 `--color-token-*`、`--codex-*` 和 `--vscode-*` 变量。
- 内置壁纸与用户自定义壁纸共用全局表面交界渐变；壁纸来源不得拥有独立的界面边缘处理。
- 适合颜色、透明度、边框、表面、文字、输入区、弹层、终端和状态色。
- 不适合创建新节点、保存用户设置或处理文件。

### C. DOM 运行时：统一交互能力

源码：`theme/static/theme.js`

- 创建月海助手节点。
- 加载打包壁纸或 IndexedDB 中的自定义壁纸。
- 保存透明度、亮度、清晰度和正文增强设置。
- 给根节点切换月海和正文增强状态类。
- 监听标题栏按钮出现，仅用于诊断当前选择器是否命中。

### D. 客户端资源注入：安装时完成

源码：`src/build-static.mjs`

- 解包并重新打包 Codex 的 `app.asar`。
- 在主页面写入外观桥和月海资源引用。
- 安装包写入主主题 CSS、运行时 JS、宠物浮层 CSS、壁纸和外观桥。
- 将壁纸与元数据复制到 `webview/moonsea/`。
- 通过定位 Codex 的外观 RPC 导出来调用官方 Appearance 动作。

这层是安装能力，不属于某张皮肤的视觉配置。不得为换一个颜色重复发明新的注入入口。

## 4. 界面分区清单

| 界面区域 | 当前稳定边界 / 入口 | 当前控制项 | 改造方式 | 新皮肤建议 |
| --- | --- | --- | --- | --- |
| 整个窗口底层 | `body::before` | 壁纸、位置、渐变、亮度、模糊 | 月海 CSS + JS 变量 | 普通使用渐变，Pro 使用图片 |
| 顶部栏 | `--codex-titlebar-tint`、`--vscode-titleBar-*`、`header.app-header-tint` | 顶部表面色与透明度 | Pro CSS 变量映射 | 窗口顶栏与主区共享令牌；主区内部线程工具栏必须透明，末端按官方工具栏高度向正文渐隐 |
| 顶部栏图标按钮 | `.draggable button[aria-label]` 等 | 图标色、按钮表面、边框、悬停、按下、焦点 | Pro CSS 选择器 | 可独立提高不透明度保证识别，不与整条顶栏绑定 |
| 左侧边栏 | `nav[aria-label]`、`--color-token-side-bar-background` | 深色遮罩、轻微模糊、顶部过渡 | Pro CSS | 允许比主区域更实；顶部与主区交界使用全局 Pro 渐变，不按壁纸分叉 |
| 右侧主区域 | `main.main-surface`、`--color-token-main-surface-primary` | 主透明表面、左缘与顶部过渡 | Pro CSS | 它是连续工作画布，不保留硬边框和环境阴影；侧栏交界横向渐隐，工具栏末端纵向渐隐 |
| 欢迎页 / 对话正文 | 主区域内部官方结构 | 继承文字与主表面 | 官方令牌 + Pro CSS | 普通内容不单独包卡片 |
| Markdown 正文增强 | Markdown 语义类片段、`.markdown`、`.prose` | 阅读底、边框、文字阴影、代码块、引用、表格 | Pro CSS + 根状态类 | 只覆盖正文，不压暗整张壁纸 |
| 输入区 / Composer | 类名包含 `composer`、`Composer` 或测试属性 | 高不透明控制表面 | Pro CSS 变量覆盖 | 必须比主区域更实，保证输入与语音按钮清晰 |
| 弹窗 / 菜单 | `[role="dialog"]`、`[role="menu"]`、Radix wrapper | 强表面、边框、24px 模糊 | Pro CSS | 使用强表面，禁止继承主区域的低透明度 |
| Tooltip | `[role="tooltip"]` | 强表面、文字、边框、18px 模糊 | Pro CSS | 保持短文本清晰，不透明度不可跟随主画布降太低 |
| 终端 / 编辑器 / 菜单 | `--vscode-*` | 背景、前景、输入、列表状态 | Pro CSS 变量映射 | 以功能可读性为先，目前基本保持高不透明 |
| 月海助手入口与面板 | `#codex-moonsea-controls` | 更新状态、壁纸设置、特效、壁纸选择、重置 | JS DOM + `assistant.css` | 普通与 Pro 完全共享 |
| 宠物通知浮层 | `data-avatar-overlay-*` | 浮层表面、文字、边框、Composer | 独立页面 CSS 注入 | 与主皮肤解耦，优先适配官方稳定 data 属性 |

## 6. 当前用户设置与换算

源码：`theme/static/theme.js`

| 设置 | 控件范围 | 默认值 | 实际换算 | 影响范围 |
| --- | --- | --- | --- | --- |
| 界面透明度 | `22..88%` | `70%` | 主区域 alpha = `clamp(0.12, 1 - transparency / 100, 0.78)` | 主区域与顶部栏 |
| 侧栏透明度 | 无独立控件 | 跟随界面透明度 | `min(0.86, mainAlpha + 0.16)` | 左侧边栏 |
| 控制表面透明度 | 无独立控件 | 跟随界面透明度 | `min(0.96, mainAlpha + 0.50)` | Composer、助手入口等 |
| 背景亮度 | `55..115%` | `94%` | `brightness / 100` | 壁纸整体 |
| 背景清晰度 | `20..100%` | `100%` | 模糊 = `(100 - sharpness) / 20`，最大 `4px` | 壁纸整体 |
| 正文增强 | 开 / 关 | 开 | 切换阅读状态类 | Markdown 阅读层 |
| 自定义壁纸 | PNG/JPEG/WebP/GIF/AVIF，最大 40MB | 无 | 保存至 IndexedDB | 只覆盖背景来源；表面系统与当前主题共用 |

设置保存在 `localStorage` 的 `codex-moonsea-theme-settings-v2`；自定义壁纸保存在 IndexedDB 的 `codex-moonsea-theme/assets`。

## 8. 选择器稳定性与风险

### 优先使用

- Codex 官方 CSS 变量：`--color-token-*`、`--codex-*`、`--vscode-*`。
- HTML 语义：`main`、`nav[aria-label]`、ARIA role。
- 官方 data 属性：`data-avatar-overlay-*`、`data-testid`。
- 月海自有 ID、class 与 `--moonsea-*` 令牌。

### 允许但必须测试

- `main.main-surface`、`.draggable`：语义明确，但类名仍可能随 Codex 改版变化。
- `[class*="electron:h-toolbar"]`、`[class*="composer"]`、`[class*="_markdownContent_"]`：属于结构探测，不是长期契约。
- VS Code 兼容变量：适合覆盖终端与编辑器，但必须在 Codex 更新后重新验收。

### 禁止

- 依赖构建生成的完整哈希类名。
- 使用 `nth-child`、深层 DOM 层级或坐标定位去猜区域。
- 一个选择器失效后静默套用半套皮肤。
- 为兼容未知旧结构同时堆多套兜底选择器。

## 9. 新皮肤制作模板

### 普通渐变壁纸

1. 在 `src/theme-catalog.mjs` 新增主题对象。
2. 配置 `mode`、`surface`、`ink`、`accent` 与真实背景渐变。
3. 由统一调色板生成器创建完整 `runtime.palette`。
4. 运行完整主题对比度与构建测试。
5. 不创建普通壁纸专属运行时分支。

### Pro 壁纸皮肤

1. 把原图放到 `assets/wallpapers/`，建议不低于 2560×1440。
2. 在 `src/wallpaper-catalog.mjs` 配置原图、预览图、构图位置、预览渐变、运行时遮罩和调色板。
3. 优先复用现有 `--moonsea-*` 表面系统；只有设计语义确实不同才新增令牌。
4. 内置与自定义壁纸只允许图片、默认位置和默认亮度不同；界面交界渐变必须归全局 Pro 表面系统。
5. 若要新增控件，先确认官方能力无法完成，再修改运行时。
6. 分别验证顶部栏、侧边栏、右侧正文、Composer、弹层、终端、设置页和宠物浮层。

## 10. 已知待收口项

### 月海助手样式存在重复所有权

- `theme/static/assistant.css` 已经定义完整助手组件。
- `theme/static/theme.css` 末段又定义了一套同名助手组件样式。
- Pro 模式下 `theme.css` 后加载，会覆盖 `assistant.css`；常规模式只使用 `assistant.css`。

根治方向：助手的结构、尺寸、交互状态统一归 `assistant.css`；Pro 主题只通过 `--moonsea-assistant-*` 或 `--moonsea-*` 令牌换肤。后续制作新皮肤时禁止继续向 `theme.css` 复制助手组件规则。

## 11. 每张皮肤的验收清单

- [ ] 普通与 Pro 都走同一个 `applyRuntimeTheme`，且桥中不存在官方 `set_theme` 颜色调用。
- [ ] 普通壁纸官网封面渐变与 Codex 实际背景渐变完全一致。
- [ ] Pro 顶部栏与右侧主区域共享连续表面令牌，没有明显横向断层。
- [ ] 侧边栏在复杂壁纸上仍能清楚阅读项目名与导航。
- [ ] Composer、菜单、弹窗、Tooltip 和终端不因主透明度降低而失去可读性。
- [ ] Markdown、代码块、引用与表格都能识别层级。
- [ ] 顶部栏按钮包含默认、hover、active、focus-visible、disabled 状态。
